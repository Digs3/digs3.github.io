Git (Distributed Version Control System)

Official Website / Tutorial

Git is the most widely used version control system. Git tracks changes to source code over time. Changes are stored in a database and can be rolled back if a mistake is made. It also enables multi-user sharing and makes deployment easier, faster, and more secure.

Git runs on the local development computer and on the server (repository) which can have multiple git repositories on it. GitHub is an example common repository, but you can install git on your own server as well.

Git stores data as snapshots. When you commit you take a picture of the project file system.
Git stores files in its database as hash values.

At any point a new branch can be created from an individual commit. This allows you to develop or experiment on this branch without affecting your master branch. Master is always the name of the main branch

Installing Git

On Windows, use git for windows.

Git Setup

Git commits are signed with your name and email address. We set these in the global config.

git config --global user.name "Daniel Dilly": set username
git config --global user.email daniel.dilly@gmail.com: set email
git config --list: list config
git help <verb>: get help

Using Git

Initializing a Project

Run git init in your project directory to initialize an empty Git repository.

This creates a hidden .git directory in your project that contains all the metadata and the database necessary for tracking the project

Use git status to check status of git. This shows what brach you are on and which files are tracked.

Add files to the staging area (tracked) to be commited with the next commit using the git add <file>

git add "*.txt" - add all text files

git add . - add all files

Create a git.ignore file to add files and folders that git will completely ignore

Saving a Project Version (Commit)

git commit -m "Initial Commit" will take the files in the staging area and lock them in and save them as a commit.

Inlucde the -a modifier to skip the staging step. Any files tracked that are changed are automatically staged. New files must still be manually added.

Use git log to show all the commits made so far.

Remote Repositories

Use git remote -v to list remote repositories

Use git remote add origin <project-url> to add a remote repository with the name 'origin'. You can add multiple remote repositories.

Use git clone <git-url> to download a project from a remote repository

Use git fetch origin to get any new changes made since the last clone or fetch. This pulls the data into the local repository but not the working directory. You will have to merge manually

Use git pull origin to fetch and merge changes made on the remote brach into the current branch.

Use git push origin master to push it up to the remote repository. Add the -u modifier to save this as the default so you can just do git push in the future.

Branching

Use git branch <BranchName> to create a new branch

Use git checkout <BranchName> to switch to a branch

Use git merge <BranchName> on branch master to merge the selected branch with it.

Use a merge tool like TorchMerge or WinMerge to help with merge conflicts.

Stashing

Use git stash to stash your working directory and git stash apply to go back to it. Include the --index modifier to the apply command to reapply the staged changes.

Use git stash list to view your stashes.

Use git stash drop to remove a stash or git stash pop to apply & drop a stash simultaneously.

3 Main States your files can reside in

  • Committed - file is safely stored in db
  • Modified - file is changed but not yet committed to db
  • Staged - file is marked as modified and ready to go into next commit snapshot

3 Main Sections of a Git project

  • Git directory - where Git stores the metadata and object db for your project
  • Working directory - a single checkout of one version of the project pulled out of db in the Git dir and placed on disk for you to modify
  • Staging area (index) - a file, often in Git dir, that stores info about what will go into next commit

Git Workflow

  1. Modify files in working dir
  2. Stage files, adding snapshots of them to your staging area
  3. Commit, takes files in staging area and stores that snapshot permanently in Git dir

Creating a Git Repository

Import an existing project or directory into Git

git init <project name>: creates the project dir, .git subdir and master branch
git add *.html: tracks the file(s)
git commit -m 'msg': commit

Clone an existing Git repository from another server

git clone <url> <dir>: clones a url into the specified directory

Use git status to determine which files are in which state.
Use the .gitignore file to list file patterns to ignore

Remote repository - version of your project hosted on the Internet or network somewhere
origin - the default name for a cloned remote repository

Branch - a lightweight movable pointer to a commit
Default branch is named "master"
There is a special pointer named "HEAD" that points to the current branch

Lightweight Tag - like a branch that doesn't change (a pointer to a specific commit)
Annotated Tag - stored as full objects in Git db (recommended)

Commands

git clone <url> Copies project from GitHub to desktop
git status Determine which files are in which state (-s for short)
git diff Shows exactly what has changed
git diff --staged Show what has changed and is staged
git add <filename> Add file to be tracked and/or staged (--all for all files)
git commit -m "<msg>" Commit (-v to show changes)
git commit -a Commit all tracked files (skip the git add)
git commit --ammend Recommit replacing the previous commit
git rm <file> Remove a file from staging area and working dir
git rm --cached <file> Remove a file from staging area but keep in workin dir
git mv <from> <to> Move/rename file
git log Show commit history (--oneline --decorate --graph --all) (--pretty=oneline)
git remote See which remote servers are configured, lists the shortnames (-v shows url urls)
git remote add <sname> Adds a new remote Git repository as an easily referencable shortname
git remote show <remote> Show more info about a remote
git remote rename <sname> Rename remote (also changes remote branch names)
git remote rm <sname> Remove remote
git fetch <remote-name> Get data from remote projects
git push <remote> <branch> Push commits back to the server (e.g., git push origin master), doesn't transfer tags
git push <remote> <tag> Push specified tag (use --tags instead of the tag name to push all tags)
git pull Automatically fetch and then merge a remote branch into current branch
git branch List branches (* indicates HEAD ptr) (-v shows last commit on each branch)
git branch --merged Lists branches already merged (--no-merged lists branches that have not been)
git branch <name> Create a new branch (pointer) at the commit you're currently on (-d to delete)
git checkout <name> Switch branch, HEAD ptr, changes working dir (-b to automatically create & switch)
git merge <branch> Merge specified branch into current branch
git tag Lists the tags
git tag -a v1.4 -m <msg> Create annotated tag
git tag v1.4 Create lightweight tag (don't supply the -a, -s, or -m option)
git tag -a v1.4 <checksum> Tag a commit that you've move passed. Specify the commit checksum (or part of it)
git show v1.4 Show tag data along with the commit tagged
